# xTest 测试用例评审功能完整实施方案

## 文档信息

- **项目名称**: xTest 测试管理平台
- **功能模块**: 测试用例评审 + 我的工作台
- **设计日期**: 2026-03-31
- **版本**: v1.0
- **技术栈**: Node.js + Express + MySQL + Socket.io

***

## 一、功能概述

### 1.1 核心功能

#### 测试用例评审模块

- 用例从"草稿"到"发布"的状态流转管理
- 完整的评审流程：提交评审 → 评审操作 → 状态变更
- 评审历史记录和审计追踪
- 实时通知机制

#### 我的工作台模块

- 用户登录后的第一着陆页
- 信息聚合与待办聚焦
- 数据概览看板
- 实时任务更新

### 1.2 业务价值

1. **提升测试质量**：通过评审机制确保用例质量
2. **提高协作效率**：实时通知和清晰的任务分配
3. **增强可追溯性**：完整的评审历史记录
4. **优化用户体验**：工作台聚合展示，减少操作路径

## 二、数据库设计方案

### 2.1 修改现有表：test\_cases

**新增字段：**

```sql
-- 添加评审相关字段到test_cases表
ALTER TABLE test_cases 
ADD COLUMN review_status ENUM('draft', 'pending', 'approved', 'rejected') DEFAULT 'draft' COMMENT '评审状态',
ADD COLUMN reviewer_id INT NULL COMMENT '评审人ID',
ADD COLUMN review_submitted_at TIMESTAMP NULL COMMENT '提交评审时间',
ADD COLUMN review_completed_at TIMESTAMP NULL COMMENT '评审完成时间',
ADD INDEX idx_review_status (review_status),
ADD INDEX idx_reviewer_id (reviewer_id);

-- 添加外键约束
ALTER TABLE test_cases
ADD CONSTRAINT fk_reviewer FOREIGN KEY (reviewer_id) REFERENCES users(id) ON DELETE SET NULL;
```

**字段说明：**

| 字段名                   | 类型        | 默认值     | 描述                                                             |
| --------------------- | --------- | ------- | -------------------------------------------------------------- |
| `review_status`       | ENUM      | 'draft' | 用例当前评审状态：draft(草稿), pending(待评审), approved(已通过), rejected(被驳回) |
| `reviewer_id`         | INT       | NULL    | 当前指定的评审人ID (关联users表)                                          |
| `review_submitted_at` | TIMESTAMP | NULL    | 提交评审的时间                                                        |
| `review_completed_at` | TIMESTAMP | NULL    | 评审完成的时间                                                        |

### 2.2 新增表：review\_records

**建表SQL：**

```sql
CREATE TABLE IF NOT EXISTS review_records (
  id INT PRIMARY KEY AUTO_INCREMENT,
  case_id INT NOT NULL COMMENT '测试用例ID',
  reviewer_id INT NOT NULL COMMENT '评审人ID',
  submitter_id INT NOT NULL COMMENT '提交人ID',
  action ENUM('submit', 'approve', 'reject', 'resubmit') NOT NULL COMMENT '操作类型',
  comment TEXT COMMENT '评审意见',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  INDEX idx_case_id (case_id),
  INDEX idx_reviewer_id (reviewer_id),
  INDEX idx_submitter_id (submitter_id),
  INDEX idx_action (action),
  INDEX idx_created_at (created_at),
  FOREIGN KEY (case_id) REFERENCES test_cases(id) ON DELETE CASCADE,
  FOREIGN KEY (reviewer_id) REFERENCES users(id) ON DELETE CASCADE,
  FOREIGN KEY (submitter_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='测试用例评审记录表';
```

**字段说明：**

| 字段名            | 类型        | 描述                                                         |
| -------------- | --------- | ---------------------------------------------------------- |
| `id`           | INT       | 主键，自增                                                      |
| `case_id`      | INT       | 关联的测试用例ID                                                  |
| `reviewer_id`  | INT       | 实际进行评审操作的用户ID                                              |
| `submitter_id` | INT       | 提交评审的用户ID                                                  |
| `action`       | ENUM      | 操作类型：submit(提交评审), approve(通过), reject(驳回), resubmit(重新提审) |
| `comment`      | TEXT      | 评审意见/备注                                                    |
| `created_at`   | TIMESTAMP | 记录创建时间                                                     |

### 2.3 数据库设计亮点

1. **状态字段使用ENUM类型**：确保数据一致性，避免无效状态
2. **双时间戳设计**：`review_submitted_at`和`review_completed_at`分别记录提交和完成时间
3. **完整的审计追踪**：`review_records`表记录所有评审操作
4. **索引优化**：为常用查询字段添加索引，提高查询性能
5. **外键约束**：保证数据一致性，支持级联删除

## 三、业务流程设计

### 3.1 评审状态流转图

```
草稿(draft) → 待评审(pending) → 已通过(approved)
                  ↓
              被驳回(rejected) → 修改后重新提审
```

### 3.2 详细业务规则

#### 3.2.1 发起提审

**触发条件：**

- 用例创建者选择评审人员（支持多选，但只有一个主评审人）
- 用例当前状态为`draft`或`rejected`

**执行操作：**

1. 系统自动将用例状态变更为"待评审"(`pending`)
2. 更新`test_cases`表的`review_status`、`reviewer_id`、`review_submitted_at`字段
3. 插入`review_records`记录（action='submit'）
4. 通过WebSocket实时通知评审人员
5. 记录操作日志到`activity_logs`表

**数据验证：**

- 评审人必须存在且为有效用户
- 用例必须存在且状态允许提交评审

#### 3.2.2 评审操作

**触发条件：**

- 评审人员查看用例详情
- 当前用户为指定的评审人
- 用例当前状态为`pending`

**执行操作：**

1. 给出评审结论：**通过** 或 **驳回**
2. 驳回时**强制要求**填写评审意见（最少20字）
3. 通过时评审意见为选填
4. 记录评审历史到`review_records`表

**数据验证：**

- 当前用户必须为指定的评审人
- 驳回时必须填写评审意见
- 评审意见最多500字

#### 3.2.3 状态流转

**通过评审：**

1. 状态变更为"已通过"(`approved`)
2. 更新`test_cases`表的`review_status`和`review_completed_at`
3. 用例可被执行
4. 通过WebSocket通知用例创建者

**驳回评审：**

1. 状态变更为"被驳回"(`rejected`)
2. 更新`test_cases`表的`review_status`和`review_completed_at`
3. 创建者需修改后重新提审
4. 通过WebSocket通知用例创建者

**重新提审：**

1. 状态再次变更为"待评审"(`pending`)
2. 更新`test_cases`表的`review_status`和`review_submitted_at`
3. 保留历史评审记录
4. 通过WebSocket通知评审人

## 四、API接口设计

### 4.1 评审模块接口

#### 4.1.1 提交评审

**接口信息：**

```
POST /api/testcases/:id/submit-review
```

**请求参数：**

```json
{
  "reviewer_id": 12,
  "comment": "请评审此用例"
}
```

**响应示例：**

```json
{
  "success": true,
  "message": "已提交评审",
  "data": {
    "case_id": 123,
    "review_status": "pending",
    "reviewer_id": 12,
    "reviewer_name": "张三",
    "submitted_at": "2026-03-31T10:30:00Z"
  }
}
```

**业务逻辑：**

1. 验证用例存在且当前状态为`draft`或`rejected`
2. 验证评审人存在且为有效用户
3. 更新`test_cases`表
4. 插入`review_records`记录
5. 通过WebSocket通知评审人
6. 记录操作日志

#### 4.1.2 执行评审

**接口信息：**

```
POST /api/testcases/:id/review
```

**请求参数：**

```json
{
  "action": "approve",
  "comment": "用例设计合理，覆盖全面"
}
```

**响应示例：**

```json
{
  "success": true,
  "message": "评审完成",
  "data": {
    "case_id": 123,
    "review_status": "approved",
    "reviewer_name": "张三",
    "reviewed_at": "2026-03-31T14:20:00Z"
  }
}
```

**业务逻辑：**

1. 验证当前用户为指定的评审人
2. 验证用例当前状态为`pending`
3. 如果action='reject'，验证comment不为空
4. 更新`test_cases`表
5. 插入`review_records`记录
6. 通过WebSocket通知用例创建者
7. 记录操作日志

            <input type="radio" name="reviewer" value="${user.id}" id="reviewer-${user.id}">
            <label for="reviewer-${user.id}">
                <span class="user-avatar">👤</span>
                <span class="user-name">${user.username}</span>

    document.getElementById('all-users-list').innerHTML = userListHtml;
}

// 选择评审人
let selectedReviewerId = null;
let selectedReviewerName = null;

function selectReviewer(userId, userName) {
    selectedReviewerId = userId;
    selectedReviewerName = userName;
    
    // 更新UI
    document.querySelectorAll('.user-item').forEach(item => {
        item.classList.remove('selected');
    });
    event.currentTarget.classList.add('selected');
}

// 确认提交评审
async function confirmSubmitReview() {
    if (!selectedReviewerId) {
        showErrorMessage('请选择评审人');
        return;
    }
    
    const caseId = window.currentEditingCaseId;
    const comment = document.getElementById('review-comment').value;
    
    try {
        showLoading('提交中...');
        
        const response = await apiRequest(`/testcases/${caseId}/submit-review`, {
            method: 'POST',
            body: JSON.stringify({
                reviewer_id: selectedReviewerId,
                comment: comment
            })
        });
        
        hideLoading();
        
        if (response.success) {
            showSuccessMessage('提交成功');
            closeSubmitReviewModal();
            
            // 刷新用例详情
            loadCaseDetail(caseId);
        } else {
            showErrorMessage(response.message || '提交失败');
        }
    } catch (error) {
        hideLoading();
        console.error('提交评审失败:', error);
        showErrorMessage('提交失败: ' + error.message);
    }
}

// 关闭提交评审对话框

    document.getElementById('submit-review-modal').style.display = 'none';
    selectedReviewerId = null;
